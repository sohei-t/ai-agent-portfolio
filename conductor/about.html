Web Claude Bridge v2

概要

Web Claude Bridge v2 は、ターミナル上で動作する Claude Code の tmux セッションを Web ブラウザからリアルタイムに操作・監視するブリッジシステムです。

SSE（Server-Sent Events）による差分ストリーミングと WebSocket による双方向コマンド送信の二重チャネルアーキテクチャを採用し、低遅延かつ効率的なリアルタイム通信を実現しています。

Overview

Web Claude Bridge v2 is a bridge system that lets you monitor and control Claude Code's tmux sessions in real time from a web browser.

It employs a dual-channel architecture combining SSE (Server-Sent Events) for diff-based output streaming and WebSocket for bidirectional command transmission, achieving low-latency and efficient real-time communication.

機能

  • セッション管理 -- 複数の tmux セッションを一覧表示・切り替え
  • リアルタイム出力 -- SSE 差分ストリーミングによるターミナル出力のライブ表示
  • コマンド送信 -- WebSocket 経由でのテキスト入力・コマンド実行
  • 承認 / 拒否 -- Claude Code の確認プロンプトに対する Approve / Deny ボタン
  • Ask Peer -- 別の Claude インスタンスへの質問機能
  • Reviewer ペイン -- Worker と Reviewer の分割表示
  • 状態自動検出 -- idle / thinking / tool_use / waiting_approval など
  • 接続状態表示 -- WS / SSE のステータスをリアルタイム表示
  • トースト通知 -- 操作結果やエラーのポップアップ通知

Features

  • Session Management -- List and switch between multiple tmux sessions
  • Real-time Output -- Live terminal display via SSE diff streaming
  • Command Input -- Send text and execute commands over WebSocket
  • Approve / Deny -- One-click response to Claude Code confirmation prompts
  • Ask Peer -- Send questions to another Claude instance
  • Reviewer Pane -- Split view for Worker and Reviewer
  • Auto State Detection -- idle / thinking / tool_use / waiting_approval
  • Connection Status -- Real-time WS / SSE status display
  • Toast Notifications -- Pop-up notifications for actions and errors

アーキテクチャ

SSE + WebSocket の二重チャネル設計により、出力ストリーミング（読み取り）とコマンド送信（書き込み）を完全に分離しています。

Browser (React + Zustand)
  |
  +-- SSE  <----  FastAPI /api/sessions/{name}/stream
  |                 |
  |            capture-pane (1s polling)
  |                 |
  |            DiffEngine (差分計算)
  |                 |
  |            SessionState 検出
  |                 (idle / thinking / tool_use / waiting_approval)
  |
  +-- WS   <--->  FastAPI /ws
                    |
               AsyncTmuxManager
                    |
               tmux send-keys / capture-pane
                    |
               Claude Code (tmux session)

なぜ二重チャネルか？

  • SSE (読み取り専用): ブラウザの自動再接続、HTTP/2 多重化、プロキシ通過性に優れる
  • WebSocket (書き込み): 低レイテンシの双方向通信、コマンドの即座な送信に最適
  • 読み取りと書き込みを分離することで、片方が切断しても他方が独立して動作

Architecture

The dual-channel design (SSE + WebSocket) cleanly separates output streaming (reads) from command input (writes).

Browser (React + Zustand)
  |
  +-- SSE  <----  FastAPI /api/sessions/{name}/stream
  |                 |
  |            capture-pane (1s polling)
  |                 |
  |            DiffEngine (diff computation)
  |                 |
  |            SessionState detection
  |                 (idle / thinking / tool_use / waiting_approval)
  |
  +-- WS   <--->  FastAPI /ws
                    |
               AsyncTmuxManager
                    |
               tmux send-keys / capture-pane
                    |
               Claude Code (tmux session)

Why Dual Channels?

  • SSE (read-only): Auto-reconnection, HTTP/2 multiplexing, excellent proxy traversal
  • WebSocket (write): Low-latency bidirectional communication, ideal for instant command dispatch
  • Separating reads from writes means one channel can survive independently if the other disconnects
